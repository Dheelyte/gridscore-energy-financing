# Stage 8 Report — Lender/DFI Analytics + Platform Admin Console

## 1. Stage name & objective
**Stage 8 — Lender/DFI analytics + platform admin.** The other two pitch
audiences: a **lender/DFI** view of portfolio-level risk across the cooperative
(PD distribution, risk-tier mix, operator concentration, and an "additional debt
capacity unlocked" estimate), a **platform-admin** console (operator onboarding +
API-key issuance, cooperative health, the active model, and audit-log search),
and — the second moat artifact — the **AUC-vs-operators network-effect chart**,
generated by **actually retraining** the model on growing operator subsets, not
hard-coded. Acceptance: lender and admin dashboards render real aggregates; the
network-effect chart comes from real retraining.

## 2. What was built
- **Network effect** (`app/services/analytics/network_effect.py`) — for
  cooperative sizes 1..N, only the first k operators' events are visible; pooled
  features are recomputed and the model **retrained**; reports held-out ROC-AUC
  (averaged over splits) + average visible history + customers covered. (ADR-0026.)
- **Cooperative analytics** (`app/services/analytics/cooperative.py`) — real
  aggregates: cooperative health counts; lender portfolio (latest score per
  customer → PD histogram, tier mix, operator concentration, **newly-bankable**
  customers from the `cooperative_lift` table, debt-capacity-unlocked heuristic);
  audit search.
- **Endpoints**: `GET /v1/analytics/portfolio` and `/v1/analytics/network-effect`
  (lender_viewer or platform_admin; the network effect is computed off the event
  loop and cached); `GET /v1/admin/health`, `/v1/admin/audit`, `/v1/admin/model`
  (platform_admin only).
- **Frontend** — a **Lender analytics** page (KPIs incl. debt-capacity, the
  network-effect line chart, PD histogram, operator-concentration bars) and a
  **Platform admin** page (health KPIs, operator onboarding + key issuance,
  active-model card, audit-log search). Role-aware nav now shows Lender/Admin
  tabs only to the right roles.

## 3. Key decisions & trade-offs
- **ADR-0026** — the network effect is measured by retraining on growing operator
  subsets. AUC gains are modest and slightly noisy (enrichment is a strong
  baseline); the **coverage curve** (avg visible history 2.2 → 12.4 months as
  operators join) is the clean structural signal, and at the shipped config the
  AUC also rises monotonically (0.693 → 0.737).
- **Real aggregates only** — every figure derives from rows; "newly bankable" and
  "debt capacity unlocked" come from the materialised `cooperative_lift` table
  (solo-reject → pooled-approve), a documented heuristic, not a guess.
- The network-effect endpoint is **cached** (first call retrains ~20 s off the
  event loop); a precompute step is a deployment option.

## 5. How to run it (from a clean checkout)
```bash
# Backend up + seeded + a model (so analytics have data):
cd backend && uv pip install -e ".[dev]"
uv run python ../scripts/train_model.py
# (start postgres+redis, alembic upgrade head, seed_demo.py — see Stage 7)
uv run uvicorn app.main:app --port 8000

cd frontend && npm run dev
#   Log in as the platform admin → "Admin" tab (health, onboarding, audit) and
#   "Lender analytics" tab (portfolio + the network-effect chart).
```

## 8. Known issues & gaps
- **AUC gains are modest / mildly noisy** per seed because enrichment is a strong
  baseline; the coverage curve is the robust monotone signal (documented honestly).
- **First network-effect call is slow** (~20 s retraining) then cached; a
  precompute-on-deploy step would remove the cold-start.
- "Debt capacity unlocked" is a **documented heuristic** ($200 representative loan
  × newly-bankable customers), not a financial model.
- Lender scope is the **whole cooperative** (a per-lender funded-operator mapping
  is future work); admin model view shows the **active bundle** (a fuller MLflow
  registry browser is deferred).
